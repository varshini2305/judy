## 1. Goal

Judy is an **LLM-as-a-judge** for pairwise question-answering: given a task spec, a
question, and two candidate answers (A, B), decide which is better. The SFT track
asks: **does fine-tuning the judge model's weights improve agreement with
ground-truth labels**, vs. the untuned base model and vs. our context-engineering
methods (prompt/policy edits)?

- Base judge model: **Gemini 3.5 Flash** (also the model being fine-tuned).
- This is a *separate* track from the context-engineering variants (V0â€“V5); here we
  change **weights**, not the prompt.

## 2. Training data

- **Source:** synthetic, generated by `gpt-5.4-nano` (`scripts/generate_judy_benchmark.py`),
  producing **objective** pairwise cases where one answer is constructed to be
  correct and the other to be a plausible failure. Ground-truth label
  (`objectively_better`) is known by construction.
- **Conversion to SFT rows:** `judy/tuning/export.py` â†’ `item_to_sft_rows()`.
  The winner is `item.correct_side()`.
- **Swap balancing:** each item is emitted in both A/B orders (the winner letter is
  flipped accordingly) to discourage position bias.
- **Sizes:** small checkpoints â€” `judy/tuning_datasets/sft/gemini35-flash/` has
  `judy_objective_checkpoint_{20,40,60,80,100}`. A representative split is on the
  order of **~56 objective cases â†’ ~112 swap-balanced train rows**, with tiny
  val/test. (Numbers vary per checkpoint.)
- **Split:** `train` / `val` / `test` JSONL per checkpoint dir.

### Training prompt + target format (the key detail)

From `judy/tuning/export.py` â†’ `build_sft_prompt()`:

```
You are a question-answering judge.

SYSTEM PROMPT:
{system_prompt}

QUESTION:
{question}

ANSWER A:
{answer_a}

ANSWER B:
{answer_b}

Return only which answer is better: A or B.
```

**Target/completion = the bare letter `A` or `B`.** There is **no chain-of-thought
or rationale** in the target, and **no evaluation policy** (`SKILL.md`) in the prompt.

## 3. Evaluation

- **Script:** `scripts/eval_tuned_judge.py` (base vs tuned, same held-out set).
- **Held-out test:** `judy/data/datasets/judy_benchmark_objective_test_100_full.jsonl`
  (100 objective cases, disjoint from training).
- **Path:** `eval_tuned_judge` â†’ `judy/eval/harness.eval_split()` â†’ `judy/judge/judge.judge_item()`,
  which builds the prompt with `build_user_prompt()` and passes the **`SKILL.md`
  policy as the system instruction**.

### Evaluation prompt + expected output format

From `judy/judge/judge.py` (`build_user_prompt` + `_OUTPUT_SPEC`), with `SKILL.md`
as the system instruction:

```
[system instruction = SKILL.md evaluation policy]

SYSTEM PROMPT (the task spec to judge against):
{system_prompt}

QUESTION:
{question}

ANSWER A:
{answer_a}

ANSWER B:
{answer_b}

Return JSON only: {"verdict":"A"|"B","margin":1-5,"rationale":"<=40 words",
"criteria":[{"name":str,"winner":"A"|"B"|"tie"}]}
```

**Expected output = a JSON object**, parsed by `_coerce_verdict()`.

- **Metrics** (`judy/metrics/metrics.py`): agreement with the label (headline),
  position-consistency (verdict stable under A/B swap), position-consistent
  agreement. **Order-swap ON** by default.

## 5. Concerns / things to validate (the review questions)

1. **Train/serve skew (primary concern).** The model is **trained** on
   `build_sft_prompt` with a **bare-letter** target and **no policy**, but
   **evaluated** on `build_user_prompt` with the **`SKILL.md` policy** and a
   **JSON** output schema. Training and inference use different prompt templates,
   different system instructions, and different output formats. *Is this misalignment
   a likely cause of the non-improvement, and is fixing it (train in the exact
   inference format) the right first step?*
2. **Label-only targets.** Targets contain no reasoning. *For a judge, should SFT
   targets include a (correct) rationale / reasoning trace rather than just the
   final letter?*
3. **Data difficulty.** Training data is **objective** cases the base model already
   solves (~92.5%), while the real headroom in our other experiments is
   **adversarial** instruction-following. *Should SFT data be drawn from cases the
   judge actually fails, not ones it already gets right?*
4. **Sample size.** ~100 train rows is small for SFT of a strong model. *Is this far
   too little to expect a gain, and/or a catastrophic-forgetting risk?*
5. **Method fit.** Labels are *verifiable* (we know the correct answer). *Is plain
   SFT the right tool here, or would preference optimization (DPO) / reinforcement
   fine-tuning (reward = agreement with the label) be more appropriate for a judge?*

## 6. Infra notes (for completeness)

- **"RLHF" status:** only a **DPO-style preference dataset is exported**
  (`judy/tuning_datasets/pref/...`, chosen/rejected pairs); there is **no RL/DPO
  trainer or reward model** in the repo yet.
- **Cloud submission:** `scripts/run_gemini_sft.py` is **prep-only** (writes GCS
  upload commands + a request stub); the actual Vertex tuning job was launched
  manually. Auth uses local `gcloud` ADC.
- **Provider portability:** an OpenAI client exists (`judy/llm/openai_client.py`).
  OpenAI supports SFT, DPO, and reinforcement fine-tuning (RFT); switching would
  only need an OpenAI chat-format exporter + `OPENAI_API_KEY` (no GCP service
  account).
